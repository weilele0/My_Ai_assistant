## 一、项目概述

### 1.1 项目简介

**My_AI_Assistant** 是一个基于 Go + Gin 构建的**全栈 AI 写作助手**，面向需要 AI 辅助写作、RAG 知识问答的个人用户和小型团队提供 SaaS 服务。

系统集成 DeepSeek 大模型进行文章生成，基于 Chroma 向量数据库 + 通义千问 Embedding 实现 RAG 智能问答，并通过 Redis Stream 实现异步任务队列，提供完整的用户体系、额度控制和后台管理能力。

### 1.2 核心能力

| 模块 | 描述 |
|------|------|
| AI 文章生成 | 输入主题，DeepSeek 大模型同步 / 异步生成 800 字左右高质量长文 |
| RAG 智能问答 | 上传文档 → 自动向量化入库 → 基于向量检索的精准问答 |
| 文档管理 | 上传 / 查看 / 删除文档，支持 txt/md 纯文本 |
| 异步任务队列 | 长文生成通过 Redis Stream 异步处理，实时轮询进度 |
| 额度管理 | 多套餐体系（免费 / 基础 / 专业 / 无限制），每日 AI & RAG 次数控制 |
| 管理后台 | 用户管理、套餐分配、系统统计、今日额度监控 |
| 用户系统 | JWT 鉴权，bcrypt 密码加密，管理员权限分级 |

### 1.3 目标用户

- **普通用户**：有 AI 写作、文档问答需求的个人用户
- **管理员**：平台运营方，负责用户管理、套餐分配和数据监控

## 三、模块设计

### 3.1 用户认证模块

#### 3.1.1 注册流程

```
前端 POST /api/v1/user/register
    ↓
JWTAuth 中间件（无需认证，直接通过）
    ↓
userHandler.Register()
    ↓
userService.Register()
    ├─ 检查用户名是否已存在
    ├─ bcrypt.GenerateFromPassword(password, DefaultCost)
    └─ userRepo.Create(user) → MySQL
```

**安全措施**：
- 密码使用 bcrypt 加密（cost=10），不可逆存储
- 用户名字段唯一索引，重复注册返回错误
- Email 字段唯一索引

#### 3.1.2 登录流程

```
前端 POST /api/v1/user/login  {username, password}
    ↓
userHandler.Login()
    ├─ userRepo.FindByUsername(username)
    ├─ bcrypt.CompareHashAndPassword(hashedPassword, password)
    └─ jwt.GenerateToken(userID, username, isAdmin)
    ↓
返回 { token, id, username, is_admin }
```

#### 3.1.3 JWT 认证中间件

- 从 `Authorization: Bearer <token>` 提取 Token
- 使用 `JWTSecret` 进行 HS256 签名验证
- Token 有效期：**24 小时**
- 验证成功后将 `user_id` / `username` / `is_admin` 写入 Gin Context
- 后续 Handler 通过 `c.Get("user_id")` 获取当前用户 ID

### 3.2 AI 生成模块

#### 3.2.1 同步生成（POST /api/v1/ai/generate）

```
请求 → JWTAuth → QuotaCheckAI（扣减 AI 额度）→ AIHandler.Generate
    ↓
aiService.GenerateContent(topic)
    ├─ 构建 System Prompt（角色：资深技术文档创作者）
    ├─ 构建 User Prompt（800字文章，含引言/核心/总结结构）
    └─ client.CreateChatCompletion("deepseek-v4-flash", messages, temperature=0.7)
    ↓
返回 { code:200, data:{ content } }
```

#### 3.2.2 异步生成（POST /api/v1/tasks/generate）

```
请求 → JWTAuth → TaskHandler.CreateAsyncGenerateTask
    ↓
taskService.CreateAsyncTask(userID, "generate", topic)
    ├─ taskRepo.Create(task{status:"pending"})
    └─ redisClient.XAdd("ai_task_stream", taskData)
    ↓
立即返回 { task_id }

---
[后台 Worker Goroutine]
redisClient.XReadGroup("ai_task_group", "worker-1", "ai_task_stream")
    ↓
processStreamMessage(msg)
    ├─ taskRepo.FindByID → status="processing"
    ├─ aiService.GenerateContent()
    ├─ taskRepo.Update(status="completed"/"failed", outputText/errorMsg)
    └─ redisClient.XAck("ai_task_stream", msgID)
```

前端通过 `GET /api/v1/tasks/:id` 轮询任务状态。

### 3.3 文档管理模块

#### 3.3.1 上传流程

```
POST /api/v1/documents  {title, content}
    ↓
documentService.UploadDocument(userID, title, "", content)
    ├─ documentRepo.Create(doc) → MySQL
    └─ Goroutine: vectorizeDocument(documentID, content)
        ├─ chunkText(content, 500)  → 按500字符分块
        ├─ embeddingService.EmbedText(chunk) → 1536维向量
        └─ chromaService.AddDocument("my_knowledge_base", docID, embedding, metadata)
```

#### 3.3.2 分块策略

- **分块大小**：500 字符（按 rune 计数，正确处理中文）
- **块 ID 格式**：`{documentID}_{chunkIndex}`（如 `12_0`、`12_1`）
- **元数据**：包含 `document_id`、`chunk_index`、`text`

### 3.4 RAG 智能问答模块

#### 3.4.1 完整流程

```
POST /api/v1/ai/rag-generate  {question, top_k?}
    ↓
JWTAuth → QuotaCheckRAG → RAGHandler.GenerateWithRAG
    ↓
ragService.GenerateWithRAG(userID, question, topK)
    │
    ├─1─ embeddingService.EmbedText(question) → queryEmbedding (1536维)
    │
    ├─2─ chromaService.SearchSimilar("my_knowledge_base", queryEmbedding, topK)
    │       → 返回最相似的 topK 条文档片段（metadata.text）
    │
    ├─3─ 构建上下文字符串
    │       【参考资料 1】
    │       {text1}
    │       【参考资料 2】
    │       {text2}
    │       ...
    │
    ├─4─ buildPrompt(question, context)
    │       → 系统提示词（基于参考资料回答，不可虚构）
    │
    ├─5─ aiService.GenerateContent(prompt) → DeepSeek 答案
    │
    └─6─ Goroutine: saveHistory(userID, question, answer, references)
            → ragHistoryRepo.Create(history, keepLimit=20)
                → 自动裁剪只保留最近 20 条
    ↓
返回 { code:200, data:{ answer, references:[text1, text2, ...] } }
```

#### 3.4.2 Prompt 设计

- **角色**：专业、严谨的知识助手
- **约束**：严格基于参考资料，不得虚构
- **结构**：先给核心结论，再分点详细说明
- **引用**：用【X】标注参考资料来源序号

### 3.5 额度管理模块

#### 3.5.1 套餐体系

| 套餐 | PlanType | 每日 AI 生成上限 | 每日 RAG 上限 | 说明 |
|------|----------|----------------|--------------|------|
| 免费版 | `free` | 5 次 | 3 次 | 新用户默认 |
| 基础版 | `basic` | 30 次 | 20 次 | 需管理员分配 |
| 专业版 | `pro` | 100 次 | 60 次 | 需管理员分配 |
| 无限制 | `unlimit` | 不限 (-1) | 不限 (-1) | 仅管理员可分配 |

#### 3.5.2 额度扣减时机

- **AI 生成**：`POST /api/v1/ai/generate` 前置中间件 `QuotaCheckAI` 立即扣减
- **RAG 问答**：`POST /api/v1/ai/rag-generate` 前置中间件 `QuotaCheckRAG` 立即扣减
- 异步任务：提交时扣减（即使未执行也算占用）
- 管理员后台操作：**不消耗额度**

#### 3.5.3 每日重置

- 用量按**自然日**统计（`date` 字段格式 `2006-01-02`）
- 每日凌晨自动重置（由 `GetOrCreate` 按当天日期查询，新一天自然无记录）
- 过期订阅自动降级为免费版

### 3.6 管理后台模块

管理员功能（需 `is_admin=true`）：

| 功能 | 接口 | 说明 |
|------|------|------|
| 用户列表 | `GET /api/v1/admin/users` | 分页展示所有用户 |
| 用户详情 | `GET /api/v1/admin/users/:id` | 查看用户 + 今日额度 |
| 设置套餐 | `POST /api/v1/admin/users/:id/plan` | 分配/变更用户套餐，支持设置到期时间 |
| 系统统计 | `GET /api/v1/admin/stats` | 用户总数、活跃数、今日调用次数、套餐分布 |
| 今日额度 | `GET /api/v1/admin/quotas/today` | 今日所有用户用量明细 |
| 订阅列表 | `GET /api/v1/admin/subscriptions` | 所有用户订阅记录 |

- **集合名称**：`my_knowledge_base`（由 DocumentService 和 RAGService 共用同一 UUID）
- **文档 ID**：`{documentID}_{chunkIndex}`（保证全局唯一）
- **向量维度**：1536（通义千问 text-embedding-v2）
- **元数据字段**：`document_id`、`chunk_index`、`text`

## 八、关键设计决策

### 8.1 为什么用 Redis Stream 而非普通 Queue？

- **消息持久化**：Stream 支持消费者组 ACK，未处理消息不会丢失
- **多 Worker 扩展**：未来可启动多个 Worker 实例（不同 consumerName）并行消费
- **消息追溯**：XACK + XREADGROUP 机制保证 Exactly-Once 消费

### 8.2 RAG 为什么用 Goroutine 异步保存历史？

- 历史记录保存不影响用户感知（异步，不阻塞返回）
- 即使保存失败也不影响 RAG 核心功能
- 自动裁剪（limit=20）控制数据库体积

### 8.3 文档向量化为什么用 Goroutine？

- 上传接口立即返回，不等待向量化完成（可能耗时数秒）
- 向量化和文章生成并行，不互相阻塞

### 8.4 Chroma 集合名称

- `DocumentService` 和 `RAGService` 统一使用 `my_knowledge_base` 集合
- 已统一，无需额外修改
